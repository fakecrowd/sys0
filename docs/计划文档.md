# sys0 · 计划文档

> 课程设计 · 一个 headless 的远程指令控制与监控平台
> 技术栈：中心服务器 / 被控端 = Go，控制台前端 = Vite + React + shadcn/ui

---

## 0. 合法使用声明

本系统仅用于**课程设计演示与教学**，所有被控端均部署在**自有或经明确授权的机器**上。
未经授权对他人计算机实施远程控制属于违法行为，本项目不为任何违法用途提供支持。
被控端启动时应有可见的运行提示与日志，不做隐藏、自启动伪装等行为。

## 1. 项目概述

### 1.1 目标
**sys0** 是一个 **中心化、headless** 的远程指令控制与监控平台：操作者通过浏览器登录 Web 控制台，对一台或多台已上线的被控端下发指令并实时查看执行结果与运行状态。被控端无图形界面，聚焦于命令执行、文件操作、进程与资源监控。

### 1.2 三类角色
- **被控端（Agent / Node）**：headless 的 Go 程序，主动连接中心服务器，阻塞接收指令、执行、回送结果，断线自动重连。
- **中心服务器（Hub）**：唯一公网入口。一侧接入被控端长连接，另一侧为 Web 控制台提供 REST + WebSocket，并在两者之间路由消息；同时负责鉴权、节点注册表、文件中转与持久化。
- **Web 控制台（Console）**：浏览器前端，操作者的唯一操作入口。

> 架构上采用「中心服务器居中转发」的三层模型：被控端与控制台都不直接互联，全部经由 Hub 中继。这样被控端只需一条出站连接即可被管理，便于穿透与统一鉴权。

## 3. 技术选型

### 3.1 中心服务器 / 被控端（Go）
- Go 1.22+
- 网络：标准库 `net`（TCP 接入）+ `nhooyr.io/websocket` 或 `gorilla/websocket`（WS 接入与前端）

- 鉴权：JWT（操作者会话）+ 预共享接入密钥（被控端注册）
- 主机指标采集：`gopsutil`
- 配置：YAML / env；日志：标准库 `slog`
- 前端嵌入：`embed` 将构建产物打包进单二进制

### 3.2 前端（Web 控制台）
- Vite + React + TypeScript
- shadcn/ui（基于 Radix + Tailwind CSS）
- 状态：`zustand` + `@tanstack/react-query`
- 路由：`react-router`

- 图表：`recharts`（资源时序 / 在线时长）
- **主题：默认深色，技术 / 控制台风格**（详见 §7.3）：等宽字体、紧凑信息密度、低饱和强调色、运维仪表盘观感

## 4. 通信协议设计

> 设计原则：**编码层与传输层解耦**。消息编码统一用 **JSON-RPC 2.0**；传输层做成**可插拔抽象**，首批实现 TCP 与 WebSocket，后续可平滑扩展（QUIC、mTLS、消息队列中转等）而不改动上层逻辑。

### 4.1 传输抽象层（Transport）
底层连接统一收敛到一组接口，被控端与服务器网关都面向接口编程：

```go
// 一条已建立的双向消息连接（已处理分帧，按"消息"收发而非字节流）
type Conn interface {
    Read(ctx context.Context) ([]byte, error)   // 读一个完整 JSON-RPC 消息
    Write(ctx context.Context, b []byte) error  // 写一个完整 JSON-RPC 消息
    RemoteAddr() string
    Close() error
}

type Dialer   interface{ Dial(ctx context.Context) (Conn, error) }   // 被控端侧
type Listener interface{ Accept(ctx context.Context) (Conn, error) } // 服务器侧
```

首批实现：
| 传输 | 分帧方式 | 说明 |
| --- | --- | --- |
| `tcp` | 4 字节大端长度前缀 + body | 解决 TCP 粘包，建议叠加 TLS |
| `ws` | WebSocket 原生消息边界 | 同端口可同时服务被控端与前端，穿透代理更友好 |

被控端通过配置 `transport: tcp|ws` 选择拨号方式；服务器可**同时**监听两种传输，接入后上层行为完全一致。前端 ↔ 服务器为 REST（登录、列表、文件元数据等同步操作）+ WebSocket（承载同一套 JSON-RPC）。

### 4.2 消息编码：JSON-RPC 2.0
三方（被控端、操作者、服务器）所有控制消息统一为 JSON-RPC 2.0，三种形态：

```jsonc
// 请求（需响应）
{ "jsonrpc": "2.0", "id": "r1", "method": "shell.run",
  "params": { "cmd": "uname -a", "timeout": 30 } }
// 成功响应（id 原样回带配对）
{ "jsonrpc": "2.0", "id": "r1", "result": { "stdout": "...", "exit": 0 } }
// 错误响应
{ "jsonrpc": "2.0", "id": "r1", "error": { "code": 4010, "message": "node offline" } }
// 通知（无 id，不需响应）：心跳、流式输出、事件
{ "jsonrpc": "2.0", "method": "emit",
  "params": { "ref": "r1", "chan": "stdout", "seq": 3, "data": "..." } }
```

### 4.3 寻址与下发：dispatch 信封
JSON-RPC 本身无寻址字段。控制台不直接寻址被控端，而是向服务器发起一个 **dispatch** 调用，由服务器完成选择、扇出、id 重映射与结果聚合：

```jsonc
// 控制台 → 服务器：把一次"对节点的调用"包进 dispatch
{ "jsonrpc": "2.0", "id": "r1", "method": "dispatch", "params": {
    "select": { "nodes": ["n_7", "n_9"] },        // 或 { "tags": ["lab-a"] } / { "all": true }
    "call":   { "method": "shell.run", "params": { "cmd": "uptime", "timeout": 30 } }
}}

// 服务器 → 控制台：按节点聚合结果
{ "jsonrpc": "2.0", "id": "r1", "result": { "items": [
    { "node": "n_7", "ok": true,  "value": { "stdout": "...", "exit": 0 } },
    { "node": "n_9", "ok": false, "error": { "code": 4010, "message": "offline" } }
] }}
```

服务器把 `call` 重新封装为标准 JSON-RPC 请求转发给目标被控端（生成内部 id → 映射回 `r1` + node），收齐后聚合。流式输出（长命令、监控采样）由被控端以 `emit` 通知逐条上送，服务器透传给控制台，最后再返回终结响应。

### 4.4 指令分级
- **Hub 方法**：服务器本地处理，不下发被控端。
- **Node 方法**：经 `dispatch` 路由到一个或多个被控端执行。

#### Hub 方法（控制台 ↔ 服务器）
| method | 说明 | params |
| --- | --- | --- |
| `dispatch` | 向选定节点下发一次 Node 调用（见 §4.3） | `{ select, call }` |
| `hub.nodes` | 在线节点列表 | `{}` |
| `hub.node` | 单节点详情 | `{ "node": "n_7" }` |
| `hub.label` | 设置节点别名 / 标签 | `{ "node": "n_7", "label": "...", "tags": [] }` |
| `hub.detach` | 主动断开某节点 | `{ "node": "n_7" }` |
| `hub.subscribe` | 订阅事件流（上下线、状态、指标推送） | `{ "topics": ["node", "metrics"] }` |

#### Node 方法（经 dispatch 下发被控端）
| method | 说明 | params |
| --- | --- | --- |
| `shell.run` | 执行命令，回送 stdout/stderr/退出码；长输出走 `emit` 流式 | `{ "cmd": "...", "cwd": "...", "timeout": 30 }` |
| `host.info` | 主机静态信息（OS、内核、CPU 型号、内存、IP、启动时长） | `{}` |
| `host.metrics` | 采集一次资源指标 | `{}` |
| `host.watch` | 开/关周期性指标推送（监控时序） | `{ "interval": 5, "enable": true }` |
| `proc.list` | 进程列表（可按名过滤） | `{ "filter": "" }` |
| `proc.signal` | 向进程发送信号 / 结束 | `{ "pid": 1234, "sig": "TERM" }` |
| `fs.ls` | 列目录 | `{ "path": "/" }` |
| `fs.stat` | 文件 / 目录元信息 | `{ "path": "..." }` |
| `fs.get` | 从被控端取文件到服务器（走文件通道） | `{ "path": "..." }` |
| `fs.put` | 推文件到被控端（走文件通道） | `{ "path": "...", "blob": "b_12" }` |
| `fs.rm` | 删除文件 / 目录 | `{ "path": "..." }` |
| `node.config` | **热更新**配置（心跳间隔、别名、传输等），免重启生效 | `{ ... }` |
| `node.reconnect` | 重建连接 | `{}` |
| `node.shutdown` | 退出被控端进程 | `{}` |

> **文件传输**不走 JSON-RPC 报文：`fs.get`/`fs.put` 仅传递元数据与一个 `blob` 句柄，实际字节经独立的文件流通道分块传输（块偏移 + sha256 校验 + 断点续传）。

## 5. 操作逻辑与交互模型

> 不采用"全局聚焦单一目标"的隐式状态，改为 **显式目标 + 工作集** 的模型——目标永远显式、可多选，服务器端不保存"当前选中"这种隐藏状态。

### 5.1 被控端接入流程
1. 启动 → 读配置（Hub 地址、`transport`、接入密钥、本机别名）。
2. 按 `transport` 选择 Dialer 拨号建连。
3. 发起握手 `node.hello`：携带接入密钥、稳定的节点指纹、主机摘要、Agent 版本与能力清单。
4. 服务器校验密钥 → 按指纹复用或新建节点身份 → 回 `result`（含 `nodeId`、心跳间隔、会话令牌）。
5. 进入消息循环：阻塞接收 JSON-RPC 请求 → 分发器按 `method` 执行 → 回响应；同时按间隔发送 `node.ping` 心跳通知。
6. 断线 → 指数退避重连，凭指纹复用同一 `nodeId`，状态从 `online`→`stale`→`offline` 流转。

### 5.2 操作者接入流程
1. 浏览器登录（用户名 + 密码）→ 服务器签发 JWT。
2. 建立 WebSocket 控制通道（握手校验 JWT），承载 JSON-RPC。
3. 调 `hub.subscribe` 订阅 `node` / `metrics` 事件流；调 `hub.nodes` 拉取初始列表。

### 5.3 指令下发流程（核心）
1. 操作者在控制台勾选一个或多个节点，形成 **工作集**（也可按标签或"全部在线"动态选择）。工作集只存在于控制台会话中。
2. 在终端 / 操作面板触发指令 → 控制台组装 `dispatch` 请求：`select`（来自工作集）+ `call`（Node 方法与参数）。
3. 服务器收到 `dispatch`：
   - 校验操作者权限与目标合法性；
   - 解析 `select` 为具体在线节点集合（离线节点直接置为 `error: offline`）；
   - 对每个目标重封装为 Node 请求并转发（内部 id 映射回 `请求id + node`）；
   - 写审计日志（谁、对哪些节点、什么方法、参数摘要、时间）。
4. 被控端执行：流式输出经 `emit` 通知逐条上送，服务器透传控制台；执行完成回终结响应。
5. 服务器聚合所有目标的终结结果，按 §4.3 的 `items` 数组返回控制台。

### 5.4 结果呈现
- **单节点**：终端区流式打印，回车即得。
- **多节点**：结果分组展示，每个节点一张卡片/一栏，带状态徽标（成功 / 失败 / 超时 / 离线）与各自输出，便于横向比对。
- 全程可在审计页回看历史指令与结果摘要。

### 5.5 权限模型（最简起步）
- 角色：`admin`（全部操作）/ `operator`（下发指令、看监控，不可改用户与节点密钥）。
- 危险指令（`shell.run`、`node.shutdown`、`fs.rm`）记审计并可在前端做二次确认。

## 6. 面向 AI Agent 的 API

> 设计目标：除了人类用 Web 控制台操作，sys0-hub 同时对外提供一套**机器友好、自描述、可被 LLM Agent 直接驱动**的 API。所有接入面共享同一个 dispatch / registry 内核，权限与审计完全一致。

### 6.1 三种接入面（同一内核）
| 接入面 | 使用者 | 鉴权 | 形态 |
| --- | --- | --- | --- |
| Web 控制台 | 人类操作者 | JWT + WebSocket | §4 / §5 已述 |
| HTTP API (v1) | 程序 / 自动化脚本 / Agent 后端 | **API Key** | REST + 同步 dispatch |
| MCP Server | LLM Agent（如 Claude） | **API Key** | MCP tools / resources |

三者最终都落到 `core/dispatch` 同一路径：选择目标 → 扇出 → 聚合 → 审计，行为与权限语义一致。

### 6.2 认证：API Key
- 独立于人类 JWT 的**机器凭据**，请求头 `Authorization: Bearer sk_...`。
- 每个 key 绑定：角色、可操作节点范围（按 tag/id 白名单）、可调用方法白名单、速率限制。
- 在控制台创建 / 吊销，入库存哈希；每次调用写审计并标注 `key id` 作为来源。

### 6.3 HTTP API（v1）
RESTful 资源 + 一个统一的同步 `dispatch` 端点：

| 端点 | 说明 |
| --- | --- |
| `GET /api/v1/nodes` | 节点列表（分页，按 `tag`/`state` 过滤） |
| `GET /api/v1/nodes/{id}` | 节点详情 |
| `POST /api/v1/dispatch` | **同步下发**，body `{ select, call }`，阻塞返回聚合 `items`（带 `timeout`） |
| `GET /api/v1/methods` | **能力发现**：返回所有 Hub/Node 方法 + JSON Schema |
| `GET /api/v1/metrics` | 监控时序查询（`node`、`from`、`to`） |
| `POST /api/v1/files` · `GET /api/v1/files/{id}` | 文件上传 / 下载 |
| `GET /api/v1/audit` | 审计查询 |
| `POST /api/v1/keys` · `DELETE /api/v1/keys/{id}` | API Key 管理（admin） |

**对 Agent 友好的关键特性**：
- **同步优先**：`POST /dispatch` 阻塞到结果聚合完成，简单 req→resp，Agent 无需管理回调；需增量时 `Accept: text/event-stream` 走 SSE 推 `emit` + 终结事件。
- **自描述**：`GET /methods` 返回每个方法的 `{ name, description, paramsSchema, resultSchema }`（JSON Schema），可**直接映射为 LLM 的 function/tool 定义**，Agent 无需硬编码。
- **结构化稳定**：统一结果信封 `{ ok, value | error }` + 稳定错误码，便于 Agent 判定与重试。
- **幂等与分页**：可选 `Idempotency-Key` 头；列表接口统一游标分页。
- **dry-run**：`dispatch` 可带 `{ "dryRun": true }`，只校验目标与权限、不实际执行，供 Agent 预检。

### 6.4 内置 MCP Server
sys0-hub 内置一个 [MCP](https://modelcontextprotocol.io) server，把 Hub/Node 能力暴露为标准 **tools / resources**，任何兼容 MCP 的 LLM 客户端可直接连接调用：

- **tools**（输入 schema 复用 §6.3 的 method schema）：`sys0_list_nodes`、`sys0_run`（在选定节点执行命令）、`sys0_host_info`、`sys0_metrics`、`sys0_fs_ls`、`sys0_fs_get` / `sys0_fs_put`、`sys0_proc_list` 等。
- **resources**：节点列表、某节点的监控时序、审计日志，作为可读资源暴露。
- **鉴权**：API Key；**内核**：仍走 `dispatch` + 审计。
- **传输**：stdio（本地代理进程）或 Streamable HTTP（远程接入）。

### 6.5 Agent 安全护栏
- 按 API Key 维度限定：方法白名单、节点范围、速率限制。
- 危险方法（`shell.run`、`node.shutdown`、`fs.rm`）对机器 key **默认禁用**，需显式开启。
- 全量审计，来源标注为 `key id`；结合 `dryRun` 让 Agent 先预检再执行。

## 7. 模块设计

仓库采用单仓多模块（monorepo）布局，三个组件以 `sys0-` 前缀命名：

```
sys0/
├── sys0-agent/     # 被控端（Go）
├── sys0-hub/       # 中心服务器（Go）
├── sys0-console/   # Web 控制台（Vite + React + shadcn）
├── docs/
└── README.md
```

### 7.1 被控端（sys0-agent）
```
sys0-agent/
├── main.go
├── transport/           # 可插拔传输：实现 Dialer/Conn
│   ├── transport.go     # 接口定义
│   ├── tcp.go           # TCP + 长度前缀分帧
│   └── ws.go            # WebSocket
├── rpc/                 # JSON-RPC 2.0 编解码 + 请求/响应/通知配对
├── conn/                # 连接生命周期：握手、心跳、断线重连
├── core/
│   ├── agent.go         # 主循环：收消息 → 分发
│   └── dispatcher.go    # 方法分发器（注册表，策略模式）
├── handler/             # 各 Node 方法处理器
│   ├── handler.go       # Handler 接口
│   ├── shell.go
│   ├── host.go          # host.info / host.metrics / host.watch
│   ├── proc.go
│   ├── fs.go            # fs.* + 文件流通道
│   └── node.go          # node.config(热更新) / reconnect / shutdown
└── util/                # 配置、日志
```

**方法分发器（策略模式）**：
```go
type Handler interface {
    Method() string
    Handle(ctx context.Context, params json.RawMessage, emit EmitFunc) (any, error)
}
// dispatcher 启动时 Register 所有 handler；收到请求按 method 查表调用。
// emit 用于流式输出（shell.run 长输出、host.watch 周期推送）。
```

### 7.2 中心服务器（sys0-hub）
```
sys0-hub/
├── main.go
├── transport/           # 与 agent 共享：Listener/Conn（tcp.go / ws.go）
├── rpc/                 # JSON-RPC 编解码 + id 路由映射
├── gateway/             # 被控端接入网关：同时挂多个 Listener
│   ├── listener.go
│   └── nodeconn.go      # 单节点连接读写循环 + 心跳判活
```

## 9. 安全设计
- 被控端接入用**预共享接入密钥**握手，服务器校验后入库（存哈希），按节点指纹复用身份。
- 操作者登录用户名 + 密码（哈希），签发 **JWT**；WebSocket 握手校验 JWT。
- 机器接入（HTTP API / MCP）用 **API Key**（存哈希），按 key 限定节点范围、方法白名单与速率；危险方法对机器 key 默认禁用，支持 `dryRun` 预检。
- 被控端 ↔ 服务器、前端 ↔ 服务器全程 **TLS**。
- 所有 `dispatch` 写**审计日志**（来源 actor=用户/key、目标集、方法、参数摘要、结果）。
- `shell.run` 做超时与输出大小上限；危险方法（`node.shutdown`、`fs.rm`）前端二次确认。
- 课程演示场景：被控端前台运行、打印明显日志，不做隐蔽化。

## 10. 开发里程碑

| 阶段 | 内容 | 产出 |

## 11. 部署与演示
- 服务器：单 Go 二进制（`go:embed` 内嵌前端），`./sys0-hub -config config.yaml` 启动。
- 被控端：单 Go 二进制，`./sys0-agent -hub <host:port> -key <接入密钥> -transport ws` 启动；可交叉编译到 Windows/Linux。
- 演示流程：启动服务器 → 浏览器登录控制台 → 启动 2~3 个被控端（混用 tcp/ws）→ 列表看到上线 → 勾选工作集下发 `shell.run`/`host.info` → 多节点结果对比 → 文件上传下载 → 监控图表与审计回看。
